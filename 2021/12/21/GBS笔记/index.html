资源的分类

进程（Process）

进程的定义

一个进程是一个在被运行的程序（Programm）

程序是被动单元，进程是主动单元

一个程序可以在多个线程中被执行（多个程序的实例）

  • 比如说：两个Microsoft Word窗口

一个系统由一组进程组成

一个进程需要资源来完成它的任务

  • 资源是CPU，储存器，输入输出设备，或者也可以是文件
  • 理想情况每个进程占有一个自己的CPU
  • 但实际上，进程必须与其他进程共享一个CPU
    • 这是操作系统的进程管理任务

每个进程都有一个自己的进程地址空间（Processadressraum）

虚拟地址空间（Virtueller Adressraum）是物理地址空间的一个抽象

地址空间装有代码和数据

不同进程的地址空间是在硬件层面与其他进程的地址空间隔离开的

进程共享一个被操作系统管理的物理储存器

连续进程（sequnzielle Prozesse）和并行进程（parallele Prozesse）

一个连续进程

  • 拥有一个控制流程/线程（Kontrollfluss/eng. Thread）和仅有一个指令计数器（Befehlszähler/eng. program counter）
  • 连续的执行程序的指令

一个并行的线程

  • 拥有多个控制流程/线程
  • 每个线程有自己的指令计数器和寄存器
  • 每个进程有自己的栈（Stack）

文件和文件系统

理念：“Everything is a file” (Unix / Linux)

通过操作系统产生的普遍的抽象

  • 文件描述器：方便输入输出的指向一个文件的引用
  • 通常来说：open，close，read，write
  • 特殊的控制：ioctl

操作系统模式（Betriessystem-Modi）

目的：从编程错误和攻击中保护操作系统

解决方法：通过工作模式来区分不同的权限

用户模式（User Mode/Space）

  • 不能直接访问硬件，所有访问必须经过操作系统
  • 不能使用特权指令（privilegierten Befehle），比如：输入输出
  • 对于访问体统代码或者数据没有或者只有只读权限
  • 只能访问虚拟地址空间

应用程序只能运行在用户模式中

系统模式（Kernel Mode/Space）

  • 有特权的模式，能够使用所有的机器指令
  • 可以使用特权代码，比如访问硬件
  • 可以访问系统代码和数据

操作系统核心运行在系统模式中

系统调用（System-Aufrufe/Systemcalls）

任务：提供一个接口，让应用程序能够使用操作系统的服务并借此能够访问硬件

解法：

  • 定义一组服务，这就是系统调用（粗略类比方法调用）
  • 系统调用通常不直接被应用程序调用，而是应用程序调用系统库，系统库调用系统调用
  • 库会被绑定给应用程序（通过Linker/Loader）

系统调用的例子

在用户模式中使用系统调用

  • 系统在启动的时候在系统模式
  • 执行用户进程的时候，系统在用户模式
  • 在想要使用操作系统服务时（例如IO），需要通过系统调用，系统调用与转入系统模式是绑定的

操作系统架构（Betriebssystem-Architekturen）

随着时间的流逝，许多不同架构的操作系统被研发了出来

概览：

  • 整体系统（Monolithische Systeme/monolithic systems）
  • 层级系统（Geschichtete Systeme/layered systems）
  • 微核系统（Mikrokernel-Systemen/microkernels）
  • 客户端服务端系统（Client-Server-Systeme）
  • 虚拟机（virtual machines）

整体系统

操作系统包含了大规模的功能

功能的概要在操作系统核心中

  • 所有的驱动都是在核心内实现的
  • 通过系统调用进入操作系统核心
  • 一个可执行程序绑定了很多程序

操作系统被当作一个程序运行

一些后果

  • 很灵活：程序可以互相调用
  • 混乱的，因为没有结构，在维护的时候会出现问题
  • 没有弹性：程序的崩溃会导致操作系统崩溃

通常：Minimale Struktur

特征:

  • 操作系统完全运行在系统模式
  • 操作系统内核有很高的进程优先级
  • 操作系统永远在内存中
  • 整体系统（经常）没有结构
  • 所以很难维护和拓展

变体：目的是更好的维护

  • 模块化（Modularität）：拓展性
    • 新的模块可以被加载，并且拓展功能性
  • 移植性（Portierbarkeit）：从软件转移到其他的硬件
    • 在操作系统中只提供基础的功能
    • 减少工作量：只有基础的功能需要移植到新的硬件结构上
  • 层级系统（Geschichtete Systeme (Layers)）
    • 根据层来分等级，通过抽象简化

微核 Mikrokerne

核心动力

理念：

  • 将操作系统拆开成独立的系统进程（eigenständige Systemprozesse）
  • 只有一个模块，即微核，在系统模式中运行
  • 小的微核：易于维护
  • 将错误隔离开

例子：L4, PikeOS, Mach, QNX, MINIX

具体实现

还提供基本机制：例如进程间交互（Prozesskommunikation (IPC)）和CPU调度（CPU-Zuteilung (Scheduling)）

子系统被当作系统服务在用户模式中实现，没有特权

例子：文件系统，储存管理或者进程管理

这样核心就是可保护的/可证明的

一些子系统也可以简单的被替换

服务进程（Serverprocess）

系统方法被当作服务进程来执行

如果一个应用程序需要一个系统服务，那它就与对应的进程沟通

服务之间的沟通并不是直接发生，而是通过操作系统核心通过IPC：客户端-服务端原则（Client-Server-Prinzip）

优点？

可以根据需要启动所需要的进程，而不是全部启动。这样就可以在一些小的平台上运行

如果一个服务进程掉了，那剩下的还可以继续运行，因为错误被子系统划分隔离（isoliert）了

缺点？

有太多的进程间交互（Inter Process Communication/IPC），所以会相对慢一点、

并行的进程程，多进程（Multiprogramming）

一个系统由一组进程组成

  • 进程可以（近似）平行的被处理
  • 存在进程之间的转换（调度 Scheduling）

近似平行的的进程处理是多进程的基础

例子：多进程系统

  • 四个有自己的控制流的不同的进程
  • 在一个单进程系统中，CPU在用一个时间点只运行一个进程
  • 随着时间的流逝，所有进程都会被执行到

I/O-bound：指一些线程要频繁的与输入输出设备交互，并不一直使用CPU，这些线程被称作I/O-bound

CPU-bound：指一些线程要一直占用CPU进行运算，很少与输入输出设备交互，这些线程被称作CPU-bound

为什么要多进程？

进程可能被CPU限制（limitiert）或者被I/O限制

  • 当一个线程要为了输入输出等它的一部分$p$，那么它的CPU负载就会出问题，在$n$个进程仅有$1-p^n$。

说白了就是用多进程抵消等候I/O的时间

例子

cat和grep各自有一个线程

如果没有多进程，那么cat读完一个文件，grep处理一个文件，相对较慢，因为在cat读文件的时候，CPU啥也没干。

如果是多进程，在cat读文件的时候，grep就可以占用CPU去处理读进来的内容，实现了更高的CPU占用率。

进程是在运行中的程序

  • 使用资源，如CPU，储存器，数据，设备等
  • 拥有一个控制流（Kontrollfluss），通过程序的指令定义

资源和控制流是两个不相关的概念

  • 区分开来！

一个进程编组（gruppieren）和管理（verwalten）资源

至今：一个进程只有一个线程

线程 Thread

定义：线程

  • 一个线程被视作一个物理处理器的抽象
  • 线程表现了一个系统的并行的执行线路
  • 一个进程拥有至少一个线程
  • 一个多线程的进程有多个流程图
  • 一个进程的线程共享一个地址空间
  • 每个线程拥有自己的指令技术器

进程 Process vs 线程 Thread

线程是进程的行为载体

线程特有的信息（线程环境）：

  • 程序技术器（program counter）
  • 寄存器的值（Registerwerte (z.B. Variablen, etc.)）
  • 管理子程序的栈（Stack zur Verwaltung von Unterprogrammaufrufen）

线程也有不同的状态

如果一个线程被拿走CPU，那么这个线程的环境安全必须被确保

  • 状态从rechnend转移到rechnenwillig或者wartend

进程编组了资源

  • 包括程序代码和数据的地址空间
  • 一个进程的所有线程共用的资源
    • 全局变量，公用的文件（offene Datei），子进程描述器，警告（Alarme）和中断（Interrupt (signal handler)）

多线程的原因

开支 Overhead

一个线程是一个轻量级的进程（此外共同的地址空间）

在一个进程的地址空间中的进程彼此交流很容易

在一个进程中创建一个线程的费用比创建一个进程要小很多

线程的转换要比进程之间的转换快

性能

多线程允许I/O密集和CPU密集的区域近似并行的执行（两个线程，一个I/O，一个CPU）

真正的线程的并行可以在多核或者多CPU上实现

进程的实现

核心数据结构：进程控制模块（Process Control Block (PCB)）

  • 包含进程管理（Prozessverwaltung (engl. process management)）信息
    • 寄存器内容（包括程序计数器 program counter，栈指针 stack pointer，状态寄存器 status register）
    • 进程状态（比如wartend，rechnend等）
      • rechnend：提供属于它的CPU
      • wartend：提供一个进程正在等候的事件
    • 优先级
    • 唯一的进程ID（Process ID (PID)）
    • 父进程ID（Parent PID (PPID)）用来标识父进程
    • 进程组ID（Process Group ID (PGID)）
  • 包含储存器管理（Speicherverwaltung (engl. memory management)）的信息
    • 指向代码段（code segement），数据段（data segement）和栈段（stack segement）的指针
    • 这些信息都包含对应段的起始位置和长度的信息
  • 包含文件管理（Dateiverwaltung (engl. file management)）
    • 根目录（Root Verzeichnis）
    • 文件解释器（File Deskriptoren）：一种数据结构，用来描述一个已经打开的文件
    • 用户（Benutzer User）和他的用户组ID（Gruppen ID (UID, GID)）

操作系统用一个进程表管理所有的进程

一个进程表是一个进程控制模块的链表（verkettete Liste）

对于不同的进程状态有不同的表

  • Run Queue（ready queue）：Linux中rechnenwillig的进程
  • Wait Queue：Linux中管理wartend的进程

这样做的原因

可以快速的找出一个目标状态的进程，也可以查出哪些进程是对应的目标状态，也可以根据需求对表中的单一目标状态的进程进行排序。省的在一张什么状态都有的表中一个个找。

僵尸进程 Zombies

父进程可以通过wait()或者waitpid()来等待子进程的终止

  • 父进程被阻塞 blockiert（被塞入wait queue），直到子进程终止
  • 子进程终止时，父进程可以询问子进程终止的原因

当子进程在父进程之前就终止了，就产生了一个僵尸进程

  • 当子进程时这样终止的话，那么就会：
    • 取消子进程的储存区域的分配
    • 把退出状态（exit status）写进子进程的进程管理模块（PCB）中，并且是能被获取到的，就是所谓的僵尸

先到先处理（FCFS First-Come-First-Served）

哪个进程先来，就先处理哪个进程，后到的进程需要排队等候被处理

不中断进程的策略

缺点：这种处理方法性能上不是最优的，因为短的进程可能后来。

先处理短的（SJF Shortest Job First）

在处理完一个进程之后，寻找剩下的进程中最短的进程，并处理它

SJF是最优的策略，如果所有进程的处理时间在一开始都是已知的

不能被用在交互系统中，因为在交互系统中不知道进程的运行时间，而且运行时间长的进程会被最后处理，如果中间来了很多比他短的进程，他就会饥饿，因为已知在被抢CPU

处理剩下时间最短的

这是一种进程中断的策略（SJF的变形）

每运行一个单位时间，就判断当前进程的余下部分是不是最短的，如果不是，就选出最短的，继续运行

死锁的条件

死锁的充分必要条件

单一进程可用的资源(Mutual Exclusion)

  • 公共的资源要么是可用的，要么是被其他进程占用的

占有和要求（就是占着碗里的看着锅里的）（Hold and Wait）

  • 已经分配了资源的进程占用了这些资源并且还要求别的资源

  • 已经分配给了一个进程的资源是不能被拿走的

环形的等待(Circular wait)

  • 每个进程都占有了一些进程，并且等待其他进程把他所需的进程放开，就形成了环形等待

要形成死锁就必须满足这四个条件。这也就是说，死锁是可以避免的，如果这四个条件的任意一个不满足

找到一个能够得到所有资源的进程，模拟这个进程释放所有资源，把这个进程移除。直到没有进程能被移除，若是还有进程留下，那就有可能死锁，反之没有

Deadlock-Avoidance

每一次资源占用之前进行一次分析，这一次的占用会不会导致死锁，如果会的话就不允许这次操作，不会的话就继续操作

Bankier-Algorithmus

大致的流程

  • 如果一个进程需要一个现在可用的资源，就尝试性的把资源分给他，然后对这个新的状态进行分析，看会不会有死锁出现。这个分析是这样的：从其他每个进程的最坏情况的资源需求出发，即其他每个进程都要他们的最大资源需求。对于这个状态进行一次Deadlock-Detection，如果不会出现死锁，那就真正的进行这次资源的分配。

但是这种方法是现实中不可行的（nicht praxistauglich）

  • 资源的需求大多情况是不清楚的也是不可估计的

显式通讯 Explizite Kommunikation

发送和接收信息：信息传递（message passing）

  • 适合在不同地址空间中的进程
  • 通过操作系统的专用的交互（Dedizierte Interaktion）
    • send (int sockfd, const char* buf, …)
    • recv (int sockfd, const char* buf, …)
  • 函数的标志可能不同，但是是在POSIX中定义了标准
  • 典型的标志：Sender/Empfänger, Nachricht
  • 通讯可以是本地的（比如：PID）也可以是和一个远端的系统（比如：IP地址，端口号）
  • 也可以使用read()/write()，因为Socket也是文件解释器

消息的构造

  • 消息头（Nachrichtenkopf (header)）：用于管理消息的一些信息，比如发送者和接收者的ID信息，消息的大小
  • 消息体（Nachrichtenkörper）：包含负载（Nutzlast/payload），即数据传输中所欲传输的实际信息

运行

  1. 进程通过send()发送消息
  2. 操作系统的消息服务传递消息
  3. 接收的进程通过recv()接收消息

有同步的和异步的通讯两种显示通讯方式

同步 vs 异步

进程之间的耦合度（Kopplungsgrad）

  • 同步（synchron）：两个进程会在消息传递的时候被同步，阻塞
  • 异步（asynchron）：发送者和接收者的解耦，不会被阻塞

消息通讯的模板

报告 Meldung（信号，不要和窄带的Linux信号混淆）

  • 从发送者向接收者的一些信息（单向的）
  • 只会传递比较少的信息
  • 例子：告知一个状态转变（例如文件成功被打印）
  • 但是：经常需要确认接收 Acknowledgment (Ack)

委托 （要求——响应 (Request – Response)）

  • 发送者和接收者的交互（双向的）
  • 例子：询问一下特定的数据，响应包含要求的数据

对于同步和异步通讯的评价

异步发送的优点

  • 对实时应用是很有用的，如果发送进程不允许阻塞
  • 允许发送者和接收者并行的处理
  • 可以用来发送事件的信号

缺点

  • 操作系统的管理开销（需要一个消息缓冲）
  • 处理错误变得很难
    • 没有直接的发送者的消息
    • 数据包（packet）丢失（缓冲满了，特别是网络通讯）
    • 数据包的重复

通常来说都是用异步通讯

  • 首先，不清楚是否接收者是否能收到或者答复多久才能计算好然后发出来
  • 可选：使用线程，来混合同步和异步消息

流 Stream

到现在我们研究的都是无连接的通讯方式，不知道消息传递的方式

一个流（Strom/Stream）是一个连接的抽象

  • 消息会在通讯的路上被缓冲
  • 被发送的信息被转换成逻辑字节流（logischer Bytestrom）
  • 字节流可以和消息无关地被处理
    • 注意：谁需要消息限制，必须自己加上去
  • 操作系统服务：连接的建立和销毁，流中的写入和读取

管道 Pipes

管道的概念（在windows和linux下）作为流的实现

  • 一个管道(“|”, Röhre)是两个通讯对象之间的一个单向流（unidirektionaler Strom）
    • 双向通讯也可以用一对管道实现
  • 一个管道允许FIFO类型的数据传输：open pipe, read, write
  • 阻塞的发生：管道满了写入会被阻塞，管道空了读取会被阻塞
  • 管道也可以用虚拟文件（virtuelle Dateien）的方式实现（记名管道 named pipes）

现实中的（匿名）管道

函数pipe()产生两个通讯端点

其中一个可以通过例如fork()给到子进程，或者在不同的线程中使用

记名的管道

给不同的进程使用

  • 进程应该能够在自己的地址空间中接收消息
  • 进程需要一个自己的给先来的，还没到的消息的一个缓冲队列

问题

  • 进程编号是会变的并且进程名字不是唯一的
  • 进程可能同时和很多的其他进程通讯

解决方案

端口

一个端口在一段时间内唯一属于一个进程

接收进程可以建立一个对于发送者特殊的端口

  • 一个自己的线程可以分配给一个端口

端口的状态可以是：offen, geschlossen, blockiert

被动进程（例如：服务器）

  • 注册一个（或者多个）端口号
  • 等待进入的连接（TCP）或者消息（UDP）
  • 一个（TCP-）端口对应着一个FIFO等待队列
    • 等待队列总结了连接愿望（Verbindungswünsche）
    • 等待队列的最大长度取决于操作系统
    • 其他的连接愿望不会被接受了，如果等待队列满了

主动进程（例如：客户端）

  • 初始化一个与被动进程的交互
  • 从操作系统中动态的获取可用的端口号
  • 从这个被动进程中就可以获得数据

用一个五元组可以标识一个通讯关系

  • Quell-IP-Adresse, Ziel-IP-Adresse, Quell-Port, Ziel-Port, Protokoll（TCP/UDP）
  • Vom BS verwendet, um eingehende Nachrichten zuzuordnen

Socket

目前的假设

  • 通过消息传送的无连接的通讯(UDP)
  • 在使用面向连接的通讯（TCP）的时候必须在两个终端之间造一个逻辑通道

两种交互方式在操作系统中的抽象：Socket（Steckdose）

  • 通讯对象之间的逻辑连接
  • 通常是双向的，但是也可以在一端关闭
  • 通常是两个终端一对一（端口和IP）
  • 也可以是广播（Broadcast (alle)，即一对所有）或者多路广播（Multicast (selektiv)，即一对多）
  • 实现了面向连接的，基于流的通讯

通过操作系统的本地集成

  • Socket解释器等于文件解释器
  • 支持同样的系统调用，如：read(), write(), close()
  • 多了一些特殊操作：创建和连接Socket（socket(), connect(), bind(), listen(), … (analog zu pipe())）
  • 控制socket（setsockopt(), getsockopt(), ioctl()）

管道，socket和进程

管道和Socket允许输入输出越过进程之间的障碍

什么会发生，如果一个和另一个进程连接中的进程终止了？

  • 管道和Socket连接会被关闭
  • 信号可以通过不同的方式发生
    • 输入的话：EOF
    • Linux信号：SIGPIPE
    • 一个错误报告：对应的文件解释器不再工作了

需要对应的错误报告

信号只会在对应的进程真的没有通讯对象之后

  • 在生成子进程的时候很重要

对于核心资源的说明

  • 所有的I/O解释器形式都能用（Standardein-/-ausgabe, Pipes, Sockets, geöffnete Dateien）

如果一个资源被打开了或者被使用了，这个资源会分配一个对应的

  • 数据结构，用来储存这个资源的信息
  • 一个引用计数器（Reference-Counter）
    • 记录这个资源过去被使用了几次
    • 在进程内部
    • 可以越过进程间的障碍
