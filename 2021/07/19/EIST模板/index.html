EIST模板

FR/NFR

Usability

  • Users need at most 5 clicks to do something.
  • Users interact with a touch-screen.
  • Users can use the software without reading the guideline.

 

Reliability

Robustness

  • The system can tolerate temperatures up to 90°C.
  • The system should be robust against cold weather.

 

Safety

  • The system can never leak user information to other users.

 

Security

  • The system should be secure even if attacked by others.

 

Performance

  • $x\%$ of $operations$ should be processed in less then 1 second
  • The system should update 60 times per second

 

Availability

  • The availability of the system is 99.7%

 

Supportability

Portability

The system should be able to run both Windows and Linux.

 

Design Goal trade-off

Functionality vs. Usability

  • the higher functionality, the more difficult to learn and thus the lower usability

 

Cost vs. Robustness

  • low cost $\rightarrow$ pay no attention to details $\rightarrow$ low Robustness

 

Cost vs. Usability

  • low cost $\rightarrow$ pay no attention to details $\rightarrow$ low Usability

 

Cost vs. Reusability

  • code for a specific problem: easy to code, low cost tests, not very reusable
  • code for general problems: additional coding and testing costs

 

Efficiency vs. Portability

  • only for one operating system: focus only on one operating system, underlying code such as Assemble, efficient
  • for serval operating systems: need additional judging conditions for different operating systems, cannot use underlying code, inefficient

 

Rapid development vs. Functionality

  • rapid development $\rightarrow$ only develop essential part $\rightarrow$ low functionality

 

Architectural style

直接用Deployment Diagram表示了, 有可以改动的地方会有详细标注

Layered architectural style

 

Tiered architectural style

Layered architectural style的变形, 每一个Layer分布在不同的节点上

 

Client server architectural style

可能存在多个Client, 每个Client可以使用多个Server提供的Service

Client和Server可能在一个节点中

 

Peer to peer architectural style

Client Server的变形, Client和Server都可以是Peer

 

Model view controller (MVC) architectural style

Model, View, Controller可能在同一个节点中

 

Repository architectural style

可能存在多个Subsystem, Subsystem和Repository可能在一个节点中

 

Design Pattern

Façade Pattern

对于所有对于Subsystem的调用, 必须通过Façade.

Bridge Pattern

将应用层面的抽象和解法层面的实践分开

Degenerated Bridge Pattern

简化的桥接模式

Proxy Pattern

通过代理对象实现对于这个对象的缓存,替代或者访问控制

Adapter Pattern

相当于转换头, 为老的系统提供新的接口, 使其能够适配新的系统

Observer Pattern

实现了一对多的依赖关系

Strategy Pattern

能够在运行中根据需求选择对应的实现
